# Requirements — Thai / English Language Toggle (Frontend Phase 8)

Source: user request to support Thai. Confirmed via clarifying question:
an EN/TH **toggle** (not Thai-only), English staying the default/fallback.

## Acceptance criteria

1. A language switcher is reachable from **every** screen, including
   `/login` (not just inside the authenticated `Layout`) — a Thai-speaking
   user shouldn't have to read English to get past the login form in the
   first place.
2. Default language is English on first visit; the chosen language
   persists across reloads/sessions (`localStorage`), per-device — no
   account-level preference, since the backend has no such field and
   nothing here requires one.
3. Every UI string this app itself authors is translated: nav labels,
   page headings, form labels/placeholders, buttons, empty-state text,
   and all **client-side** validation messages (the ones mirroring
   backend rules before a round-trip, e.g. "Name is required.").
4. **Explicit limitation, not silently glossed over**: error messages
   that come from the backend verbatim (e.g. "barcode is already in use",
   "Insufficient stock: current balance is 5") are shown in whatever
   language the API returns — English only, since `grocery-api` has no
   i18n of its own. Switching the frontend to Thai does not translate
   these. This matches the project's established pattern of surfacing
   backend gaps rather than working around them client-side.
5. Switching language is instant (no reload, no flash of missing text) and
   applies to already-rendered pages (dashboard stats, cart lines, etc.),
   not just to pages navigated to after the switch.

## Out of scope (this phase)

- Any language beyond English/Thai.
- Translating data the user themselves entered (product names, notes on
  stock transactions, etc.) — that's user content, not UI copy.
- Formatting numbers/dates/currency per-locale beyond what's already
  correct today (฿ prefix, `YYYY-MM-DD` dates) — not requested, and
  changing date *format* risks breaking the existing local-date handling
  that was deliberately built to avoid a UTC-shift bug.
